\chapter{Wir wollen verstehen, warum robotik-uno-q ROS2 braucht}

\kapitelziel{
Du kannst erklären, an welche Grenzen das selbstgebaute System
stößt und warum ROS2 die natürliche nächste Stufe ist.
}
\kapitelstruktur

\section{Situation}

Das System ist jetzt vollständig:
\texttt{sensor\_bridge.py} liest vom Arduino,
\texttt{SensorBus} verteilt die Daten,
\texttt{motor\_bridge.py} steuert die Motoren.

\begin{lstlisting}[style=text]
hinderniserkennung.ino
    |  Serial /dev/arduino
    v
sensor_bridge.py  -->  SensorBus  -->  navigation_callback()
                                  -->  logger_callback()
                                  -->  motor_bridge.py
                                            |
                                       Serial /dev/arduino
                                            v
                                  differentialantrieb.ino
\end{lstlisting}

Es funktioniert. Warum also ROS2?

\begin{aufgabeBox}
Identifiziere die vier Grenzen des aktuellen Systems und
verstehe, welches Problem ROS2 jeweils löst.
\end{aufgabeBox}

\section{Die vier Grenzen des selbstgebauten Systems}

\textbf{1. Kein zweiter Rechner.}
Alles läuft auf einem Raspberry Pi. Sobald ein zweiter
Computer -- Laptop zum Debuggen, Server für schwere
KI-Berechnungen -- hinzukommt, muss die gesamte
Kommunikation neu geschrieben werden.
\emph{ROS2 löst das: Discovery und Kommunikation über Netzwerk
sind eingebaut.}

\textbf{2. Kein Simulator.}
Den Roboter testen bedeutet: Roboter rausstellen, starten,
warten. Ohne Hardware kein Test.
\emph{ROS2 löst das: Gazebo simuliert den Roboter vollständig --
\texttt{sensor\_bridge.py} weiß nicht, ob es echten oder
simulierten HC-SR04-Sensor liest.}

\textbf{3. Keine Zeitstempel.}
\texttt{SensorBus} übergibt Abstandswerte -- aber wann wurden
sie gemessen? Bei 100\,Hz Sensor und 10\,Hz Navigation
entstehen Synchronisierungsprobleme.
\emph{ROS2 löst das: Jede Nachricht hat einen Header mit Timestamp
und Frame-ID.}

\textbf{4. Kein Standard-Visualisierungswerkzeug.}
Debugging heißt: \texttt{print()}-Zeilen lesen.
\emph{ROS2 löst das: rviz2 zeigt Sensordaten, Trajectories
und Koordinatensysteme live in 3D.}

\begin{merksatzBox}
Das selbstgebaute System ist kein Fehler -- es ist Lernen.
Wer Pipes, Serial-Kommunikation und Pub-Sub selbst
implementiert hat, versteht ROS2 sofort: Es löst dieselben
Probleme, nur auf industriellem Niveau.
\end{merksatzBox}

\begin{robotikBox}
Jede Einschränkung oben entspricht einer ROS2-Komponente:
Netzwerkkommunikation $\to$ DDS/rmw.
Simulation $\to$ Gazebo + ros2\_control.
Zeitstempel $\to$ std\_msgs/Header.
Visualisierung $\to$ rviz2.
Das sind keine optionalen Extras -- sie sind der Grund,
warum ROS2 in professionellen Robotikprojekten Standard ist.
\end{robotikBox}

% ── Kapitel 26 ────────────────────────────────────────────────
\chapter{Wir bauen den ROS2-Node-Graph im Kopf}

\kapitelziel{
Du kannst den zukünftigen ROS2-Stack von robotik-uno-q
mit Nodes, Topics und Services beschreiben.
}
\kapitelstruktur

\section{Situation}

Bevor du ROS2 installierst und konfigurierst, baust du
das Zielbild. In Band~2 wirst du jeden dieser Blöcke
implementieren -- es hilft, jetzt zu wissen, wohin die Reise geht.

\begin{aufgabeBox}
Zeichne den ROS2-Node-Graph für robotik-uno-q Stufe~2
und benenne jeden Node, Topic und Service.
\end{aufgabeBox}

\begin{lstlisting}[style=text]
ROS2-Stack robotik-uno-q (Stufe 2):

[arduino_bridge_node]        [hinderniserkennung.ino]
    |  liest Serial /dev/arduino        (auf Arduino)
    |
    +--> Topic /sensor/links   (sensor_msgs/Range)
    +--> Topic /sensor/rechts  (sensor_msgs/Range)
    |
    v
[navigation_node]
    |  subscribt auf /sensor/links + /sensor/rechts
    |  berechnet Fahrbefehl
    |
    +--> Topic /cmd_vel  (geometry_msgs/Twist)
    |
    v
[motor_node]               [differentialantrieb.ino]
    |  subscribt auf /cmd_vel           (auf Arduino)
    |  sendet Befehle ueber Serial
    |
    \--> Service /motor/stopp (std_srvs/Trigger)

[logger_node]
    |  subscribt auf /sensor/links + /cmd_vel
    |  schreibt in stufe2_ros2/logs/
\end{lstlisting}

\begin{lstlisting}[style=text]
Vergleich: eigenes System vs. ROS2

SensorBus.publish()         -> ros2 topic pub
navigation_callback()       -> create_subscription(Twist, ...)
motor_bridge.sende()        -> create_publisher(Twist, ...)
/tmp/sensor_kanal  (FIFO)   -> /sensor/links (DDS Topic)
\end{lstlisting}

\begin{merksatzBox}
Die Konzepte sind dieselben.
Die Implementierung ist professioneller.
Wer den eigenen Bus versteht, versteht ROS2 sofort.
\end{merksatzBox}

# 5. Git konfiguriert

# 7. ROS2-Container testen

\section{Band 2: Python für Robotik}

In Band~2 wird \texttt{stufe2\_ros2/} zu einem echten
ROS2-Paket. Du ersetzt \texttt{SensorBus} durch
\texttt{rclpy}-Publisher, \texttt{sensor\_bridge.py}
wird zum \texttt{sensor\_bridge\_node}, und am Ende
fährt der Roboter autonom.

\begin{lstlisting}[style=python]
# Band 2 -- arduino_bridge_node.py (Vorschau)
import rclpy
from rclpy.node import Node
from sensor_msgs.msg import Range

class ArduinoBridgeNode(Node):
    def __init__(self):
        super().__init__('arduino_bridge')
        self.pub_rechts = self.create_publisher(
            Range, '/sensor/rechts', 10)
        self.pub_links  = self.create_publisher(
            Range, '/sensor/links',  10)
        self.timer = self.create_timer(0.1, self.lese_arduino)

    def lese_arduino(self):
        # Liest von /dev/arduino, published Range-Nachrichten
        pass
\end{lstlisting}

\section{Band 3: ROS2 vertieft}

Band~3 zeigt den vollständigen Stack:
SLAM (Karte erstellen), Nav2 (autonom navigieren),
\texttt{micro\_ROS} (Arduino direkt in ROS2),
und Deployment auf dem echten Roboter.

\begin{merksatzBox}
Die Robotik-Werkstatt ist kein Vorwissen -- sie ist das Fundament.
Linux, Git, Docker und Serial-Kommunikation sind
keine Hilfsmittel. Sie sind das Handwerkszeug,
mit dem Robotikentwickler täglich arbeiten.
Wer sie beherrscht, bewegt sich in ROS2 wie zu Hause.
\end{merksatzBox}

\begin{robotikBox}
\textbf{robotik-uno-q nach Band~3:}
Fährt autonom, erstellt eine Karte seiner Umgebung (SLAM),
navigiert zu einem Zielpunkt (Nav2), kommuniziert über
\texttt{micro\_ROS} ohne Serial-Bridge, startet nach
dem Einschalten vollautomatisch -- und der gesamte Code
liegt versioniert auf \texttt{github.com/wollbaer1971/robotik-uno-q}.
\end{robotikBox}
