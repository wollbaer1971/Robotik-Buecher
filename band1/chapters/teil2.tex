% ── Kapitel 3 ────────────────────────────────────────────────
\chapter{Wir wollen eine Datei speichern und wiederfinden}

\kapitelziel{
Du lernst, wie Linux Dateien und Verzeichnisse organisiert,
navigierst das Dateisystem sicher und findest dich im
geklonten Projektordner zurecht.
}
\kapitelstruktur

\section{Situation}

Du hast gerade das Repository \texttt{robotik-uno-q} von GitHub
geklont. Im Terminal steht:

\begin{lstlisting}[style=bash]
git clone https://github.com/wollbaer1971/robotik-uno-q
\end{lstlisting}

Jetzt starrst du auf eine leere Eingabeaufforderung.
Wo bist du eigentlich gerade? Wo liegt der Ordner?
Und warum zeigt \texttt{robotik-uno-q} überhaupt nichts an,
wenn du das tippst?

\begin{aufgabeBox}
Navigiere zum geklonten Projekt, erforsche seine Struktur
und lerne, dich im Linux-Dateisystem sicher zu bewegen.
\end{aufgabeBox}

\section{Erster Lösungsversuch}

Vielleicht tippst du einfach \texttt{robotik-uno-q}.
Linux antwortet:
\begin{lstlisting}[style=text]
bash: robotik-uno-q: command not found
\end{lstlisting}

Natürlich. Ein Verzeichnisname ist kein Befehl.

\begin{problemBox}
Linux kennt keine Laufwerksbuchstaben wie \texttt{C:\textbackslash}.
Alles beginnt mit einem einzigen Wurzelverzeichnis: \texttt{/}.
Wer das nicht versteht, wird auf Anhieb keine Fehlermeldung
einordnen können.
\end{problemBox}

\begin{wissenBox}
Wir brauchen: den Aufbau des Linux-Dateisystems, die Befehle
\texttt{pwd}, \texttt{ls}, \texttt{cd} und das Konzept des
\enquote{aktuellen Verzeichnisses}.
\end{wissenBox}

\begin{haubeBox}
\textbf{Warum ein einziges Wurzelverzeichnis?}

Unix wurde in den 1970er Jahren entworfen, als Computer
noch ohne Festplatten anliefen -- oder mit mehreren, die
unterschiedliche Technologien nutzten. Statt Laufwerksbuchstaben
zu vergeben, entschied Ken Thompson: Es gibt einen einzigen
Baum. Alles wird dort \enquote{eingehängt} (gemountet) --
eine USB-Festplatte, eine Netzwerkfreigabe, ein Prozessor-Interface.

Das Ergebnis: \texttt{/dev/ttyUSB0} ist eine Datei.
\texttt{/proc/cpuinfo} ist eine Datei. Der Arduino am USB-Port
ist eine Datei. Wer \enquote{alles ist eine Datei} verstanden hat,
versteht Linux.
\end{haubeBox}

\section{Der pragmatische Weg}

\begin{lstlisting}[style=bash]
pwd                      # Wo bin ich gerade?
ls                       # Was ist hier?
ls -la                   # Alles, auch versteckte Dateien
cd ~/robotik-uno-q       # In das geklonte Projekt wechseln
ls                       # Projektstruktur erkunden
\end{lstlisting}

\begin{lstlisting}[style=text]
robotik-uno-q/
├── stufe1_mcu/          Arduino C++ Sketches
│   ├── differentialantrieb/
│   ├── hinderniserkennung/
│   └── linienfolger/
├── stufe2_ros2/         Noch leer -- wird in diesem Buch gefuellt
├── stufe3_wifi/         Band 3
├── docs/                Schaltplaene und Bilder
│   └── pics/
├── atopile/             KiCad PCB-Design
└── README.md
\end{lstlisting}

\begin{loesungBox}
\textbf{Schritt 1:} Orientierung verschaffen.
\begin{lstlisting}[style=bash]
pwd
# /home/ros
cd ~/robotik-uno-q
pwd
# /home/ros/robotik-uno-q
\end{lstlisting}
\textbf{Schritt 2:} Projektstruktur erkunden.
\begin{lstlisting}[style=bash]
ls -la
cat README.md
cat stufe1_mcu/differentialantrieb/differentialantrieb.ino | head -20
\end{lstlisting}
\textbf{Schritt 3:} Merke dir diesen Pfad -- er ist deine Basis.
\begin{lstlisting}[style=bash]
echo "export ROBOTIK=~/robotik-uno-q" >> ~/.bashrc
source ~/.bashrc
cd $ROBOTIK    # ab jetzt immer so
\end{lstlisting}
\end{loesungBox}

\begin{projektBox}
\textbf{Bisher:} \texttt{stufe1\_mcu/} mit drei Arduino-Sketches vorhanden.
\texttt{stufe2\_ros2/} und \texttt{stufe3\_wifi/} noch leer.\\
\textbf{Heute:} Dateisystem verstanden, Projekt geklont und erkundet.
\textbf{Nächstes Kapitel:} Erste Python-Datei in \texttt{stufe2\_ros2/}.
\end{projektBox}

\begin{merksatzBox}
\texttt{pwd} zeigt, wo du bist. \texttt{cd} bewegt dich.
\texttt{ls} zeigt, was da ist. \texttt{\textasciitilde} ist immer
dein Home-Verzeichnis. Diese vier Konzepte reichen für
80\,\% aller Navigation im Terminal.
\end{merksatzBox}

\begin{fehlerBox}
\begin{itemize}
  \item \textbf{Verzeichnisname als Befehl:} \texttt{robotik-uno-q}
        gibt Fehler. Richtig: \texttt{cd robotik-uno-q}.
  \item \textbf{Leerzeichen im Pfad:} \texttt{cd mein projekt}
        scheitert. Entweder Anführungszeichen oder
        Unterstriche statt Leerzeichen.
\end{itemize}
\end{fehlerBox}

\begin{uebungBox}
\textbf{Aufgabe:} Navigiere in
\texttt{stufe1\_mcu/hinderniserkennung/} und zeige den Inhalt
der \texttt{.ino}-Datei an. Wechsle dann zurück zum
Projektstamm mit einem einzigen Befehl.
\end{uebungBox}

\begin{quizBox}
\begin{enumerate}
  \item Was gibt \texttt{pwd} zurück?
  \item Was bedeutet \texttt{cd ..}?
  \item Wie navigierst du direkt zum Projektstamm, egal
        wo du gerade bist?
\end{enumerate}
\textbf{Antworten:} 1.~Den absoluten Pfad des aktuellen Verzeichnisses.
2.~Eine Ebene nach oben (zum Elternverzeichnis).
3.~\texttt{cd \$ROBOTIK} (nach obiger Einrichtung) oder
\texttt{cd \textasciitilde/robotik-uno-q}.
\end{quizBox}

\begin{robotikBox}
Das Gerät \texttt{/dev/ttyUSB0} -- über das später der Arduino
kommuniziert -- ist eine Datei im Linux-Dateisystem.
Lesen von dieser Datei bedeutet: Daten vom Arduino empfangen.
Schreiben auf diese Datei bedeutet: Befehle senden.
Wer das Dateisystem versteht, versteht damit auch,
warum die Kommunikation mit dem Mikrocontroller so einfach ist.
\end{robotikBox}

% ── Kapitel 4 ────────────────────────────────────────────────
\chapter{Wir wollen ein Programm starten}

\kapitelziel{
Du kannst Python-Skripte aus dem Terminal starten und verstehst,
warum Linux nicht automatisch in deinem Projektordner sucht.
}
\kapitelstruktur

\section{Situation}

\texttt{stufe2\_ros2/} liegt leer da. Zeit für die erste Python-Datei.
Du öffnest \texttt{nano sensor\_bridge.py}, schreibst drei Zeilen,
speicherst -- und tippst dann \texttt{sensor\_bridge.py}.

\begin{lstlisting}[style=text]
bash: sensor_bridge.py: command not found
\end{lstlisting}

Du hast die Datei gerade erst erstellt und Linux findet sie nicht?

\begin{aufgabeBox}
Starte das erste Python-Skript in \texttt{stufe2\_ros2/}
und verstehe, warum der bloße Dateiname nicht ausreicht.
\end{aufgabeBox}

\begin{problemBox}
Stell dir vor, der Dateiname ist wie ein Zettel auf dem Schreibtisch.
Linux weiß aber nicht, dass es auf diesem Schreibtisch nach
Programmen suchen soll. Es sucht nur an fest definierten Orten.
Deshalb reicht der Dateiname allein nicht -- du musst Linux
sagen, \emph{wo} die Datei liegt.
\end{problemBox}

\begin{haubeBox}
\textbf{Warum existiert der PATH überhaupt?}

Sicherheit. Wenn das aktuelle Verzeichnis automatisch im PATH wäre,
könnte jemand eine Datei namens \texttt{ls} in deinen Projektordner
legen -- und beim nächsten \texttt{ls}-Befehl würde \emph{diese}
Datei ausgeführt, nicht das echte \texttt{ls}.
Das ist keine Theorie: Es war ein realer Angriff in frühen
Unix-Systemen. Die Lösung: Das aktuelle Verzeichnis (\texttt{.})
ist bewusst \emph{nicht} im PATH.
\end{haubeBox}

\begin{wissenBox}
Wir brauchen: den PATH, den Shebang (\texttt{\#!/usr/bin/env python3}),
Ausführungsrechte (\texttt{chmod +x}) und den Unterschied zwischen
\texttt{python3 sensor\_bridge.py} und \texttt{./sensor\_bridge.py}.
\end{wissenBox}

\section{Der pragmatische Weg}

\begin{lstlisting}[style=bash]
cd ~/robotik-uno-q/stufe2_ros2

# Methode 1: Interpreter explizit angeben (empfohlen)
python3 sensor_bridge.py

# Methode 2: Als ausfuehrbare Datei (braucht Shebang + chmod)
chmod +x sensor_bridge.py
./sensor_bridge.py     # ./ = "suche hier, nicht im PATH"
\end{lstlisting}

\begin{lstlisting}[style=python]
#!/usr/bin/env python3
# stufe2_ros2/sensor_bridge.py -- Version 0.1
# Verbindet Arduino-Serial mit Python. Noch nicht implementiert.
print("[sensor_bridge] Gestartet.")
print("[sensor_bridge] Warte auf Arduino-Verbindung...")
\end{lstlisting}

\begin{loesungBox}
\begin{lstlisting}[style=bash]
cd ~/robotik-uno-q/stufe2_ros2
nano sensor_bridge.py   # Shebang in Zeile 1 nicht vergessen!
chmod +x sensor_bridge.py
./sensor_bridge.py
# [sensor_bridge] Gestartet.
# [sensor_bridge] Warte auf Arduino-Verbindung...
\end{lstlisting}
\end{loesungBox}

\begin{projektBox}
\textbf{Bisher:} Projektstamm bekannt, Dateisystem verstanden.\\
\textbf{Heute:} \texttt{stufe2\_ros2/sensor\_bridge.py} erstellt
und ausgeführt.\\
\textbf{Nächstes Kapitel:} Vollständige Verzeichnisstruktur aufbauen.
\end{projektBox}

\begin{merksatzBox}
\texttt{python3 sensor\_bridge.py} -- immer.
\texttt{./sensor\_bridge.py} -- nur wenn Shebang und
\texttt{chmod +x} gesetzt. Das \texttt{./} ist kein Fehler,
sondern Absicht: \enquote{Suche \emph{hier}, nicht im PATH.}
\end{merksatzBox}

\begin{fehlerBox}
\begin{itemize}
  \item \textbf{command not found:} Kein \texttt{./} und nicht im PATH.
  \item \textbf{Permission denied:} \texttt{chmod +x} vergessen.
  \item \textbf{bad interpreter:} Shebang-Zeile falsch (z.\,B. Windows-Zeilenende).
\end{itemize}
\end{fehlerBox}

\begin{robotikBox}
In ROS2 werden Nodes genau so gestartet: \texttt{ros2 run paket node}.
Dahinter steckt derselbe Mechanismus -- ROS2 kennt die installierten
Pakete und findet die ausführbaren Dateien über seinen eigenen PATH,
der durch \texttt{source install/setup.bash} gesetzt wird.
\end{robotikBox}

% ── Kapitel 5 ────────────────────────────────────────────────
\chapter{Wir wollen Dateien erstellen, kopieren, verschieben und löschen}

\kapitelziel{
Du kannst die Verzeichnisstruktur von \texttt{stufe2\_ros2/}
professionell aufbauen und Dateien sicher verwalten.
}
\kapitelstruktur

\section{Situation}

\texttt{sensor\_bridge.py} liegt allein in \texttt{stufe2\_ros2/}.
Bald kommen Konfigurationsdateien, Log-Ausgaben und mehrere
Python-Module dazu. Ohne Struktur wird das unübersichtlich --
und nach drei Wochen findet niemand mehr etwas.

\begin{aufgabeBox}
Baue die vollständige Verzeichnisstruktur für
\texttt{stufe2\_ros2/} auf, die später das ROS2-Paket tragen wird.
\end{aufgabeBox}

\begin{problemBox}
\texttt{rm} löscht sofort und endgültig -- kein Papierkorb, keine
Rückfrage. Wer ohne Nachdenken aufräumt, kann Stunden Arbeit
in Sekunden vernichten.
\end{problemBox}

\section{Die Zielstruktur}

\begin{lstlisting}[style=text]
stufe2_ros2/
├── src/
│   └── sensor_bridge.py     (hierher verschieben!)
├── config/
│   └── sensor_config.yaml   (entsteht in Kapitel 8)
├── launch/
│   └── roboter.launch.py    (entsteht in Band 2)
├── logs/                    (Laufzeit-Logs, in .gitignore)
└── tests/
\end{lstlisting}

\begin{loesungBox}
\begin{lstlisting}[style=bash]
cd ~/robotik-uno-q/stufe2_ros2

# Struktur anlegen
mkdir -p src config launch logs tests

# sensor_bridge.py an den richtigen Ort
mv sensor_bridge.py src/

# Pruefen
ls -la
ls src/

# logs/ aus Git ausschliessen (Logs gehoeren nicht ins Repo)
echo "stufe2_ros2/logs/" >> ../.gitignore
cat ../.gitignore
\end{lstlisting}
\end{loesungBox}

\begin{projektBox}
\textbf{Bisher:} \texttt{sensor\_bridge.py} in \texttt{stufe2\_ros2/}.\\
\textbf{Heute:} Vollständige Ordnerstruktur mit \texttt{src/},
\texttt{config/}, \texttt{launch/}, \texttt{logs/}, \texttt{tests/}.\\
\textbf{Nächstes Kapitel:} Skript im Hintergrund betreiben.
\end{projektBox}

\begin{merksatzBox}
\texttt{mkdir -p pfad/zu/ordner} erstellt alle Ebenen auf einmal.
\texttt{mv datei ziel/} verschiebt ohne Kopie -- das Original ist weg.
Vor jedem \texttt{rm} einmal \texttt{ls} -- um sicherzustellen,
was wirklich gelöscht wird.
\end{merksatzBox}

\begin{fehlerBox}
\begin{itemize}
  \item \textbf{mv ohne -r bei Ordnern:} Schlägt fehl.
        Für Ordner: \texttt{mv ordner/ ziel/} (klappt ohne \texttt{-r}).
  \item \textbf{rm -rf logs/ aus dem falschen Verzeichnis:}
        Immer erst \texttt{pwd} prüfen.
\end{itemize}
\end{fehlerBox}

\begin{robotikBox}
Ein ROS2-Paket hat eine vorgeschriebene Struktur:
\texttt{package.xml}, \texttt{setup.py}, \texttt{resource/},
\texttt{launch/}. Die Struktur, die du gerade aufgebaut hast,
entspricht exakt der ROS2-Paket-Konvention.
Wenn Band 2 beginnt, ist \texttt{stufe2\_ros2/} bereits
ROS2-kompatibel strukturiert.
\end{robotikBox}

% ── Kapitel 6 ────────────────────────────────────────────────
\chapter{Wir wollen wissen, was auf dem Computer gerade läuft}

\kapitelziel{
Du verstehst, was ein Prozess ist, kannst laufende Programme
anzeigen und gezielt beenden.
}
\kapitelstruktur

\section{Situation}

Der Roboter soll bald gleichzeitig Sensordaten lesen,
Motor-Befehle berechnen und mit dem Arduino kommunizieren.
Auf dem Arduino passiert das mit Hardware-Interrupts.
Auf dem Linux-Rechner passiert das mit Prozessen.

Bevor du \texttt{sensor\_bridge.py} im Hintergrund startest
und gleichzeitig mit dem Arduino-Sketch interagierst,
brauchst du ein Grundverständnis: Was ist da eigentlich
\enquote{am Laufen}?

\begin{aufgabeBox}
Starte \texttt{sensor\_bridge.py} im Hintergrund, finde seinen
Prozess und beende ihn gezielt.
\end{aufgabeBox}

\begin{wissenBox}
\textbf{Drei Konzepte, die du unterscheiden musst:}

\textbf{Programm:} Eine Datei auf der Festplatte.
\texttt{sensor\_bridge.py} ist ein Programm -- es liegt still da.

\textbf{Prozess:} Ein laufendes Programm. Wenn du
\texttt{python3 sensor\_bridge.py} tippst, erschafft Linux
einen Prozess -- das Programm wird lebendig, bekommt
Speicher, CPU-Zeit und eine eindeutige Nummer (PID).

\textbf{Thread:} Ein Ausführungsstrang \emph{innerhalb} eines Prozesses.
Ein Prozess kann viele Threads haben. In \texttt{sensor\_bridge.py}
werden später Sensor-Lesen und Navigation als parallele Threads
laufen -- ohne dass zwei separate Prozesse nötig sind.
\end{wissenBox}

\begin{haubeBox}
\textbf{Warum gibt es Prozesse überhaupt?}

Ein früher Computer lief immer nur \emph{ein} Programm gleichzeitig.
Das war verschwenderisch: Wenn ein Programm auf eine Festplatte
wartete, tat der Prozessor nichts. Die Lösung: Das Betriebssystem
wechselt blitzschnell zwischen vielen Programmen hin und her --
so schnell, dass es sich \enquote{gleichzeitig} anfühlt.
Jedes dieser Programme ist ein Prozess. Linux hat typischerweise
200--400 Prozesse gleichzeitig -- auch wenn du nur \texttt{nano} nutzt.
\end{haubeBox}

\section{Der pragmatische Weg}

\begin{lstlisting}[style=bash]
# sensor_bridge.py im Hintergrund starten
cd ~/robotik-uno-q/stufe2_ros2
python3 src/sensor_bridge.py &
# Ausgabe: [1] 12345  <- PID des Prozesses

# Laufende Prozesse anzeigen
ps aux | grep sensor_bridge
# ros  12345  0.0  0.1 ... python3 src/sensor_bridge.py

# Prozess beenden
kill 12345

# Falls er nicht reagiert:
kill -9 12345
\end{lstlisting}

\begin{lstlisting}[style=bash]
# Interaktive Prozess-Ansicht (q zum Beenden)
top
htop   # komfortabler: apt install htop
\end{lstlisting}

\begin{loesungBox}
\begin{lstlisting}[style=bash]
# 1. Starten
python3 ~/robotik-uno-q/stufe2_ros2/src/sensor_bridge.py &

# 2. Finden
ps aux | grep sensor_bridge
# Merke PID (z.B. 14823)

# 3. Beenden
kill 14823

# Sicherer: Prozessname statt PID
pkill -f sensor_bridge.py
\end{lstlisting}
\end{loesungBox}

\begin{projektBox}
\textbf{Bisher:} Projektstruktur aufgebaut.\\
\textbf{Heute:} Prozesse verstanden, Skript im Hintergrund
gestartet und kontrolliert beendet.\\
\textbf{Nächstes Kapitel:} Fehlermeldungen lesen und debuggen.
\end{projektBox}

\begin{merksatzBox}
PID = Process ID. \texttt{kill} sendet ein Signal: standardmäßig
SIGTERM (\enquote{bitte beende dich sauber}). \texttt{kill -9}
ist der Notausschalter: sofortige Beendigung ohne Aufräumen --
erst wenn \texttt{kill} ohne \texttt{-9} nicht wirkt.
\end{merksatzBox}

\begin{fehlerBox}
\begin{itemize}
  \item \textbf{kill ohne Rechte:} Fremde Prozesse kann nur root beenden.
  \item \textbf{kill -9 zu früh:} Offene Dateien (Logs, Datenbankverbindungen)
        können korrumpiert werden.
  \item \textbf{\texttt{\&} vergessen:} Ohne \texttt{\&} blockiert das Skript
        das Terminal -- du kannst nichts anderes eingeben.
\end{itemize}
\end{fehlerBox}

\begin{robotikBox}
\texttt{ros2 node list} zeigt alle ROS2-Nodes -- das sind Python-
oder C++-Prozesse mit einer speziellen Schnittstelle.
\texttt{ros2 node kill /sensor\_node} beendet einen Node sauber
über das ROS2-Lifecycle-System -- das ist dasselbe wie
\texttt{kill}, aber mit Aufräumlayer davor.
\end{robotikBox}

% ── Kapitel 7 ────────────────────────────────────────────────
\chapter{Wir wollen Fehlermeldungen verstehen}

\kapitelziel{
Du kannst Linux-Fehlermeldungen einordnen, systematisch
debuggen und das Ergebnis auf das robotik-uno-q-Projekt
übertragen.
}
\kapitelstruktur

\section{Situation}

\texttt{sensor\_bridge.py} soll jetzt tatsächlich Daten vom
Arduino lesen -- dafür importierst du \texttt{pyserial}.
Du startest das Skript. Rot. Fehlermeldung.

\begin{lstlisting}[style=text]
Traceback (most recent call last):
  File "src/sensor_bridge.py", line 3, in <module>
    import serial
ModuleNotFoundError: No module named 'serial'
\end{lstlisting}

Das ist kein Rückschlag. Das ist Information.

\begin{aufgabeBox}
Lerne, Fehlermeldungen systematisch zu lesen, und behebe
den Import-Fehler in \texttt{sensor\_bridge.py}.
\end{aufgabeBox}

\begin{wissenBox}
Wir brauchen: Traceback lesen von unten nach oben,
Exit-Codes, \texttt{stderr} vs. \texttt{stdout},
\texttt{pip3 install} und Logs.
\end{wissenBox}

\begin{haubeBox}
\textbf{Warum gibt es zwei Ausgabe-Kanäle?}

Unix-Programme haben von Anfang an zwei getrennte Ausgaben:
\textbf{stdout} (Standard Output) für normale Ergebnisse,
\textbf{stderr} (Standard Error) für Fehlermeldungen.

Warum getrennt? Damit du beides unabhängig umleiten kannst:
\begin{lstlisting}[style=bash]
python3 sensor_bridge.py > sensor.log 2>/dev/null
# stdout -> sensor.log, stderr ignorieren
\end{lstlisting}
In Robotik-Systemen schreiben Sensordaten auf stdout
(weiterverarbeiten) und Fehlermeldungen auf stderr
(ins Log-System). Das ist kein Zufall -- es ist Design.
\end{haubeBox}

\section{Traceback lesen -- von unten nach oben}

\begin{lstlisting}[style=text]
Traceback (most recent call last):       <- Beginn des Tracebacks
  File "src/sensor_bridge.py", line 3   <- Wo im Code?
    import serial                        <- Welche Zeile?
ModuleNotFoundError: No module named 'serial'
                                         ^--- HIER ANFANGEN ZU LESEN
\end{lstlisting}

Die letzte Zeile nennt das eigentliche Problem.
Alles darüber zeigt den Pfad dorthin.

\begin{loesungBox}
\textbf{Schritt 1:} Fehlermeldung verstehen.
\texttt{ModuleNotFoundError: No module named 'serial'}
bedeutet: \texttt{pyserial} ist nicht installiert.

\textbf{Schritt 2:} Paket installieren.
\begin{lstlisting}[style=bash]
pip3 install pyserial
\end{lstlisting}

\textbf{Schritt 3:} Exit-Code prüfen.
\begin{lstlisting}[style=bash]
python3 src/sensor_bridge.py
echo $?    # 0 = Erfolg, alles andere = Fehler
\end{lstlisting}

\textbf{Schritt 4:} Stderr von Stdout trennen.
\begin{lstlisting}[style=bash]
python3 src/sensor_bridge.py 2> stufe2_ros2/logs/fehler.log
cat stufe2_ros2/logs/fehler.log
\end{lstlisting}
\end{loesungBox}

\begin{projektBox}
\textbf{Bisher:} Projektstruktur + erster Prozess.\\
\textbf{Heute:} \texttt{pyserial} installiert, Traceback-Lesen gelernt,
Fehlerlog in \texttt{stufe2\_ros2/logs/} eingerichtet.\\
\textbf{Nächstes Kapitel:} YAML-Konfiguration für Sensor-Parameter.
\end{projektBox}

\begin{merksatzBox}
Traceback immer von \textbf{unten nach oben} lesen.
Die letzte Zeile ist das eigentliche Problem.
Exit-Code 0 = Erfolg. Jeder andere Wert = Fehler.
\texttt{echo \$?} direkt nach dem Befehl.
\end{merksatzBox}

\begin{fehlerBox}
\begin{itemize}
  \item Fehlermeldung nur halb lesen -- die wichtige Information
        steht meist am Ende.
  \item \texttt{ModuleNotFoundError} heißt nicht, dass das Modul
        nicht existiert -- vielleicht ist es für eine andere
        Python-Version installiert.
        Check: \texttt{python3 --version} und \texttt{pip3 --version}.
\end{itemize}
\end{fehlerBox}

\begin{robotikBox}
ROS2-Nodes schreiben strukturierte Logs:
\begin{lstlisting}[style=bash]
[INFO] [sensor_node]: Sensor initialisiert.
[WARN] [sensor_node]: Abstand unter Schwellenwert.
[ERROR] [sensor_node]: Serial-Port nicht gefunden.
\end{lstlisting}
Diese Log-Niveaus entsprechen exakt dem stdout/stderr-Konzept:
INFO geht an den normalen Kanal, ERROR an den Fehler-Kanal.
\end{robotikBox}
