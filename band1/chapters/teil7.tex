\chapter{Wir wollen zwei Programme miteinander sprechen lassen}

\kapitelziel{
Du verstehst, wie Prozesse Daten austauschen, und implementierst
die Serial-Brücke zwischen hinderniserkennung.ino
und sensor\_bridge.py.
}
\kapitelstruktur

\section{Situation}

\texttt{hinderniserkennung.ino} läuft auf dem Arduino und
sendet alle 100\,ms eine Zeile über Serial:

\begin{lstlisting}[style=text]
R=87 cm  L=124 cm
R=86 cm  L=123 cm
R=41 cm  L=122 cm
Bremsen
R=15 cm  L=120 cm
Hindernis rechts -- links ausweichen
\end{lstlisting}

Diese Daten soll \texttt{sensor\_bridge.py} empfangen,
parsen und weiterreichen -- an Navigation, Logging, Dashboard.

\begin{aufgabeBox}
Implementiere \texttt{sensor\_bridge.py} so, dass es
die Serial-Daten vom Arduino liest und über eine Named Pipe
(FIFO) an einen zweiten Prozess weitergibt.
\end{aufgabeBox}

\begin{wissenBox}
Wir brauchen: Pipes, Named Pipes (FIFOs), das Lesen von
Serial-Daten mit pyserial und das Flush-Problem bei Pipes.
\end{wissenBox}

\begin{haubeBox}
\textbf{Warum gibt es in Linux überhaupt Pipes?}

Unix wurde für Systeme mit wenig Speicher entworfen.
Statt ein riesiges Programm zu schreiben, das alles kann,
kombiniert man viele kleine -- jedes tut genau eine Sache gut.
Die Pipe ist das Verbindungsstück zwischen ihnen.

\texttt{cat daten.txt | grep "Hindernis" | wc -l}
-- drei Programme, eine Pipeline. Dieses Prinzip heißt
\textit{Unix Philosophy}. Es ist 50 Jahre alt und noch
heute der Kern moderner Systemarchitektur.
ROS2-Topics sind Pipes mit Netzwerkfähigkeit und Typsicherheit.
\end{haubeBox}

\section{sensor\_bridge.py -- die echte Implementierung}

\begin{lstlisting}[style=python]
#!/usr/bin/env python3
# stufe2_ros2/src/sensor_bridge.py -- Version 0.3
# Liest Serial-Daten von hinderniserkennung.ino

import serial, sys, re, yaml
from pathlib import Path

CONFIG = Path(__file__).parent.parent / "config" / "sensor_config.yaml"

def lade_config():
    with open(CONFIG) as f:
        return yaml.safe_load(f)

def parse_zeile(zeile):
    # Parst "R=87 cm  L=124 cm" -> (87, 124) oder None
    m = re.match(r'R=(\d+)\s+cm\s+L=(\d+)\s+cm', zeile.strip())
    if m:
        return int(m.group(1)), int(m.group(2))
    return None

def main():
    config  = lade_config()
    port    = config['serial'].get('port', '/dev/arduino')
    baud    = config['serial']['baudrate']
    timeout = config['serial']['timeout_s']

    print(f"[sensor_bridge] {port} @ {baud} Baud")
    try:
        ser = serial.Serial(port, baud, timeout=timeout)
        for zeile in ser:
            text = zeile.decode('utf-8', errors='replace').strip()
            ergebnis = parse_zeile(text)
            if ergebnis:
                r, l = ergebnis
                # SENSOR-Zeilen auf stdout fuer Pipe-Weiterleitung
                print(f"SENSOR {r:4d} {l:4d}")
                sys.stdout.flush()  # Pflicht bei Pipe!
            else:
                print(f"INFO  {text}", file=sys.stderr)
    except serial.SerialException as e:
        print(f"FEHLER: {e}", file=sys.stderr)
        sys.exit(1)

if __name__ == "__main__":
    main()
\end{lstlisting}

\begin{loesungBox}
\textbf{Mit simuliertem Arduino testen (kein Hardware nötig):}
\begin{lstlisting}[style=bash]
# Terminal 1: Arduino simulieren

    sys.stdout.flush()
    time.sleep(0.2)
" > /tmp/arduino_sim.fifo &

# Named Pipe anlegen
mkfifo /tmp/sensor_kanal

# Terminal 2: sensor_bridge lesen (nicht ausfuehren -- nur Konzept)
# python3 src/sensor_bridge.py < /tmp/sensor_kanal
\end{lstlisting}
\end{loesungBox}

\begin{projektBox}
\textbf{Bisher:} Sensor-Bridge hatte nur Platzhalter.\\
\textbf{Heute:} Echte Serial-Kommunikation -- parst das
Arduino-Protokoll \texttt{R=XX cm  L=YY cm} direkt.\\
\textbf{Nächstes Kapitel:} Daten per Pub-Sub verteilen.
\end{projektBox}

\begin{merksatzBox}
\texttt{sys.stdout.flush()} in Pipe-Anwendungen ist Pflicht.
Ohne explizites Flush puffert Python -- der nachfolgende
Prozess wartet ewig auf Daten, die längst berechnet wurden.
\end{merksatzBox}

\begin{robotikBox}
Das Parsing in \texttt{parse\_zeile()} macht genau das,
was in Band~2 ein ROS2-Subscriber-Callback tut:
Rohdaten empfangen, validieren, in ein Datenobjekt konvertieren.
Der Code ändert sich minimal -- er bekommt nur einen ROS2-Rahmen.
\end{robotikBox}

% ── Kapitel 22 ────────────────────────────────────────────────
\chapter{Wir wollen Nachrichten statt direkter Befehle senden}

\kapitelziel{
Du implementierst Publish-Subscribe für robotik-uno-q:
Sensor-Daten fließen vom Arduino zu Navigation und Logger
ohne direkte Kopplung zwischen den Modulen.
}
\kapitelstruktur

\section{Situation}

\texttt{sensor\_bridge.py} liefert Abstandswerte.
Navigation, Logger und Dashboard sollen alle empfangen.
Drei direkte Verbindungen würden den Code eng koppeln --
jede neue Komponente erfordert Änderungen an sensor\_bridge.

\begin{aufgabeBox}
Implementiere einen internen Message-Bus für robotik-uno-q,
der Sensordaten an beliebig viele Abnehmer verteilt.
\end{aufgabeBox}

\begin{analogie}
Publish-Subscribe ist wie ein Radiosender. Der Sender
schickt ins Leere -- er weiß nicht, wer zuhört.
Navigation, Logger und Dashboard schalten sich ein,
wenn sie wollen. Der Sender ändert sich nie.
\end{analogie}

\begin{lstlisting}[style=python]
# stufe2_ros2/src/bus.py -- Interner Message-Bus
from collections import defaultdict
import threading

class SensorBus:
    def __init__(self):
        self._subs = defaultdict(list)
        self._lock = threading.Lock()

    def subscribe(self, topic, callback):
        with self._lock:
            self._subs[topic].append(callback)

    def publish(self, topic, nachricht):
        with self._lock:
            callbacks = list(self._subs[topic])
        for cb in callbacks:
            cb(nachricht)

bus = SensorBus()

# ── Subscriber ───────────────────────────────────────────────
def navigation_callback(msg):
    r, l = msg['rechts'], msg['links']
    minimum = min(r, l)
    if minimum < 25:
        print(f"  NAV: STOPP   (min={minimum} cm)")
    elif minimum < 50:
        print(f"  NAV: LANGSAM (min={minimum} cm)")
    else:
        print(f"  NAV: FREI    ({r}/{l} cm)")

def logger_callback(msg):
    import time
    ts = time.strftime("%H:%M:%S")
    with open('stufe2_ros2/logs/sensor.csv', 'a') as f:
        f.write(f"{ts},{msg['rechts']},{msg['links']}\n")

bus.subscribe('/sensor/abstand', navigation_callback)
bus.subscribe('/sensor/abstand', logger_callback)

# ── Publisher ────────────────────────────────────────────────
def veroeffentliche(rechts_cm, links_cm):
    bus.publish('/sensor/abstand', {
        'rechts': rechts_cm,
        'links':  links_cm,
    })
\end{lstlisting}

\begin{projektBox}
\textbf{Heute:} Pub-Sub-Bus für Sensordaten. Navigation
und Logger subscriben unabhängig voneinander.\\
\textbf{Konzept:} Identisch zu ROS2-Topics.
In Band~2 wird \texttt{SensorBus} durch echte
\texttt{rclpy}-Publisher ersetzt.
\end{projektBox}

\begin{merksatzBox}
Topic-Namen beginnen mit \texttt{/} und beschreiben den Inhalt,
nicht den Sender: \texttt{/sensor/abstand}, nicht
\texttt{/bridge\_zu\_navigation}. Publisher kennen ihre
Subscriber nicht -- und das ist Absicht.
\end{merksatzBox}

\begin{robotikBox}
Dieser Bus wird in Band~2 zu:
\begin{lstlisting}[style=python]
self.publisher = self.create_publisher(
    Range, '/sensor/abstand', 10)
\end{lstlisting}
\texttt{Range} ist der ROS2-Standardtyp für Abstandssensoren.
Der Rest des Codes bleibt fast identisch.
\end{robotikBox}
