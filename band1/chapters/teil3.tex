% ── Kapitel 8 ────────────────────────────────────────────────
\chapter{Wir wollen eine Konfigurationsdatei bearbeiten}

\kapitelziel{
Du kannst Textdateien im Terminal bearbeiten und verstehst das
YAML-Format, das in ROS2 überall eingesetzt wird.
}
\kapitelstruktur

\section{Situation}
Der ROS2-Node startet, aber die Kamera-Auflösung stimmt nicht.
Die Einstellung steckt in einer YAML-Datei -- die musst du öffnen
und ändern.

\begin{aufgabeBox}
Öffne eine Konfigurationsdatei im Terminal, verstehe ihr Format
und speichere eine Änderung.
\end{aufgabeBox}

\begin{problemBox}
Grafische Editoren sind auf Robotern oft nicht verfügbar.
Wer nur GUI-Editoren kennt, steht vor einem verschlossenen System.
\end{problemBox}

\begin{wissenBox}
Wir brauchen: \texttt{nano} als einfachen Terminal-Editor, das
YAML-Format und den Umgang mit \texttt{.bashrc}.
\end{wissenBox}

\section{Der pragmatische Weg}

\begin{lstlisting}[style=bash]
# nano: einfachster Terminal-Editor
nano kamera_config.yaml

# Steuerung in nano:
# Strg+O : Speichern
# Strg+X : Beenden
# Strg+K : Zeile ausschneiden
# Strg+U : Einfuegen
# Strg+W : Suchen
\end{lstlisting}

\begin{lstlisting}[style=yaml]
# kamera_config.yaml -- YAML-Beispiel
kamera:
  aufloesung:
    breite: 1920
    hoehe: 1080
  bildrate: 30
  format: "RGB8"

roboter:
  name: "MobileRobot-1"
  max_geschwindigkeit: 0.5
  sicherheitsabstand: 0.25
\end{lstlisting}

\begin{loesungBox}
\begin{lstlisting}[style=bash]
# Datei sichern, dann bearbeiten
cp kamera_config.yaml kamera_config.yaml.bak
nano kamera_config.yaml
# Aenderung vornehmen: bildrate: 30 -> bildrate: 60
# Strg+O, Enter (speichern)
# Strg+X (beenden)

# Aenderung pruefen
diff kamera_config.yaml kamera_config.yaml.bak
\end{lstlisting}
\end{loesungBox}

\begin{merksatzBox}
YAML ist einrückungsbasiert -- Leerzeichen statt Tabs!
Ein falsches Leerzeichen macht die ganze Datei ungültig.
Kommentare beginnen mit \texttt{\#}. Zeichenketten brauchen
meist keine Anführungszeichen -- es sei denn, sie enthalten Sonderzeichen.
\end{merksatzBox}

\begin{fehlerBox}
\begin{itemize}
  \item \textbf{Tabs in YAML:} YAML erlaubt keine Tabs als Einrückung.
  \item \textbf{Falsche Einrückung:} \texttt{yaml.scanner.ScannerError} ist oft ein Einrückfehler.
  \item \textbf{nano vergessen:} \texttt{Strg+C} beendet nano \textbf{nicht} -- \texttt{Strg+X} schon.
\end{itemize}
\end{fehlerBox}

\begin{uebungBox}
Erstelle mit \texttt{nano} eine Datei \texttt{roboter.yaml} mit mindestens
drei Konfigurationsparametern (Name, Geschwindigkeit, Sensoren als Liste).
Validiere sie mit \texttt{python3 -c "import yaml; yaml.safe\_load(open('roboter.yaml'))"}.
\end{uebungBox}

\begin{quizBox}
\begin{enumerate}
  \item Wie speichert man in nano?
  \item Darf YAML Tabs zur Einrückung verwenden?
  \item Was bedeutet \texttt{\#} in einer YAML-Datei?
\end{enumerate}
\textbf{Antworten:} 1. Strg+O, dann Enter. 2. Nein, nur Leerzeichen.
3. Kommentarzeichen -- Rest der Zeile wird ignoriert.
\end{quizBox}

\begin{haubeBox}
\textbf{vim}: Wer häufig auf Servern arbeitet, lohnt sich vim zu lernen.
Grundprinzip: Modus-basiert. \texttt{i} = Eingabemodus, \texttt{Esc} = Befehlsmodus,
\texttt{:wq} = Speichern und Beenden. Einmal gelernt, extrem schnell.
\end{haubeBox}

\begin{robotikBox}
ROS2-Parameter werden in YAML-Dateien gespeichert und beim Start
eines Nodes übergeben:
\begin{lstlisting}[style=bash]
ros2 run my_robot sensor_node --ros-args --params-file kamera_config.yaml
\end{lstlisting}
Konfiguration von Code zu trennen ist eines der Kernprinzipien
professioneller Robotikentwicklung.
\end{robotikBox}

% ── Kapitel 9 ────────────────────────────────────────────────
\chapter{Wir wollen mehrere Befehle automatisch ausführen}

\kapitelziel{
Du kannst Shell-Skripte schreiben, die Abläufe automatisieren,
und verstehst Variablen, Bedingungen und Schleifen in Bash.
}
\kapitelstruktur

\section{Situation}
Du musst nach jedem Roboter-Start fünf Befehle manuell eintippen:
ROS2 laden, Workspace sourcen, drei Nodes starten. Das kostet Zeit
und führt zu Tippfehlern.

\begin{aufgabeBox}
Schreibe ein Shell-Skript, das den gesamten Startvorgang
automatisiert.
\end{aufgabeBox}

\begin{problemBox}
Ein Shell-Skript ist keine normale Textdatei. Es braucht einen
Shebang, Ausführungsrechte und eine korrekte Syntax.
\end{problemBox}

\section{Der pragmatische Weg}

\begin{lstlisting}[style=bash]
#!/bin/bash
# start_roboter.sh -- Roboter-Startskript

# Variablen
WORKSPACE=~/robotik/ros2_ws
ROS_DISTRO=jazzy

# Umgebung laden
source /opt/ros/${ROS_DISTRO}/setup.bash
source ${WORKSPACE}/install/setup.bash

echo "ROS2 geladen: $ROS_DISTRO"

# Bedingung: Workspace vorhanden?
if [ -d "$WORKSPACE" ]; then
    echo "Workspace gefunden: $WORKSPACE"
else
    echo "FEHLER: Workspace nicht gefunden!"
    exit 1
fi

# Nodes starten
ros2 launch my_robot robot.launch.py &
echo "Roboter gestartet."
\end{lstlisting}

\begin{loesungBox}
\begin{lstlisting}[style=bash]
# Skript erstellen und ausfuehrbar machen
nano start_roboter.sh
chmod +x start_roboter.sh

# Testen
./start_roboter.sh

# Debugging: Jeden Befehl ausgeben
bash -x start_roboter.sh
\end{lstlisting}
\end{loesungBox}

\begin{merksatzBox}
Jedes Bash-Skript beginnt mit \texttt{\#!/bin/bash}. Variablen werden
mit \texttt{\$} abgerufen. Leerzeichen um \texttt{=} sind verboten:
\texttt{VAR=wert} ist korrekt, \texttt{VAR = wert} ist ein Fehler.
\end{merksatzBox}

\begin{fehlerBox}
\begin{itemize}
  \item \textbf{Leerzeichen um =:} \texttt{NAME = "ROS"} gibt Fehler. Richtig: \texttt{NAME="ROS"}
  \item \textbf{Windows-Zeilenenden:} Skripte aus Windows haben \texttt{\textbackslash r\textbackslash n}. Fix: \texttt{dos2unix skript.sh}
\end{itemize}
\end{fehlerBox}

\begin{uebungBox}
Schreibe ein Skript, das:
1. Prüft ob Python3 installiert ist (\texttt{which python3}).
2. Bei Erfolg die Python-Version ausgibt.
3. Bei Fehler eine Fehlermeldung zeigt und mit Exit-Code 1 beendet.
\end{uebungBox}

\begin{quizBox}
\begin{enumerate}
  \item Was ist ein Shebang?
  \item Wie ruft man eine Bash-Variable ab?
  \item Wie prüft man in Bash, ob eine Datei existiert?
\end{enumerate}
\textbf{Antworten:} 1. Die erste Zeile (\texttt{\#!/bin/bash}), die den Interpreter angibt.
2. Mit vorangestelltem \$, z.\,B. \texttt{\$VARIABLE}. 3. Mit \texttt{if [ -f "\$DATEI" ]}.
\end{quizBox}

\begin{haubeBox}
\textbf{Pipes und Umleitungen kombiniert:}
\begin{lstlisting}[style=bash]
ros2 topic list | grep sensor | wc -l
\end{lstlisting}
Dieser Einzeiler: listet alle Topics, filtert nach "sensor",
zählt die Treffer. Pipes verbinden Programme zu einer Verarbeitungskette --
das Unix-Philosophie-Prinzip: viele kleine Programme, die je eine Sache gut machen.
\end{haubeBox}

\begin{robotikBox}
ROS2 Launch-Dateien in Python sind mächtiger als Shell-Skripte --
aber Shell-Skripte bleiben nützlich für Umgebungs-Setup,
Backup-Routinen und Deployment-Automatisierung auf dem Roboter.
\end{robotikBox}

% ── Kapitel 10 ────────────────────────────────────────────────
\chapter{Wir wollen ein Programm regelmäßig starten}

\kapitelziel{
Du kannst Prozesse automatisch starten lassen -- beim Systemstart
oder nach einem Zeitplan.
}
\kapitelstruktur

\section{Situation}
Der Roboter soll nach dem Einschalten automatisch seinen Sensor-Node
starten, ohne dass jemand manuell einloggt.

\begin{aufgabeBox}
Richte den automatischen Start eines Programms beim Hochfahren ein.
\end{aufgabeBox}

\begin{problemBox}
Cron und systemd sind zwei verschiedene Systeme mit unterschiedlicher
Syntax. Der falsche Ansatz führt zu schwer debuggbaren Problemen.
\end{problemBox}

\section{Cron -- Zeitgesteuerte Aufgaben}

\begin{lstlisting}[style=bash]
# Crontab bearbeiten
crontab -e

# Syntax: Minute Stunde Tag Monat Wochentag Befehl
# Jeden Tag um 3:30 Uhr:
30 3 * * * /home/ros/backup.sh >> /home/ros/backup.log 2>&1

# Jede Stunde:
0 * * * * python3 /home/ros/sensor_check.py

# Bei Systemstart (@reboot):
@reboot sleep 10 && /home/ros/start_roboter.sh
\end{lstlisting}

\section{systemd -- Dienste beim Systemstart}

\begin{lstlisting}[style=text]
# /etc/systemd/system/roboter.service
[Unit]
Description=Roboter Sensor Node
After=network.target

[Service]
User=ros
WorkingDirectory=/home/ros/robotik
ExecStart=/home/ros/start_roboter.sh
Restart=on-failure
RestartSec=5

[Install]
WantedBy=multi-user.target
\end{lstlisting}

\begin{lstlisting}[style=bash]
# Dienst aktivieren
sudo systemctl enable roboter.service
sudo systemctl start roboter.service
sudo systemctl status roboter.service
journalctl -u roboter.service -f   # Logs live
\end{lstlisting}

\begin{loesungBox}
Für den Roboter-Autostart: systemd ist die bessere Wahl.
Es verwaltet Abhängigkeiten (z.\,B. erst nach Netzwerk starten),
überwacht den Prozess und startet ihn bei Absturz neu.
\end{loesungBox}

\begin{merksatzBox}
Cron für geplante, regelmäßige Aufgaben. Systemd für permanente
Dienste, die beim Start laufen sollen und stabil bleiben müssen.
Der Roboter-Node gehört zu systemd.
\end{merksatzBox}

\begin{quizBox}
\begin{enumerate}
  \item Was macht \texttt{@reboot} in einem Crontab-Eintrag?
  \item Wie zeigt man den Status eines systemd-Dienstes?
  \item Was bedeutet \texttt{Restart=on-failure}?
\end{enumerate}
\textbf{Antworten:} 1. Der Befehl wird nach jedem Systemstart ausgeführt.
2. Mit \texttt{systemctl status name.service}.
3. Der Dienst wird bei einem Absturz automatisch neu gestartet.
\end{quizBox}

\begin{robotikBox}
Auf einem mobilen Roboter ist systemd unverzichtbar: Der ROS2-Stack
soll nach dem Einschalten selbstständig starten, überwacht werden
und bei Absturz neu starten -- ohne Eingriff. Das ist der Übergang
von \enquote{Laboraufbau} zu \enquote{autonomem System}.
\end{robotikBox}
