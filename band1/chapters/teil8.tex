\chapter{Wir wollen ein USB-Gerät unter Linux finden}

\kapitelziel{
Du kannst USB-Geräte unter Linux identifizieren, den
richtigen Gerätepfad finden und stabile Gerätezuordnungen
über udev-Regeln einrichten.
}
\kapitelstruktur

\section{Situation}
Du schließt deinen Arduino Uno an den Raspberry Pi an.
Python sagt: \texttt{serial.SerialException: [Errno 2] No such
file or directory: '/dev/ttyUSB0'}. Vielleicht ist es
\texttt{ttyACM0}? Oder \texttt{ttyUSB1}?

\begin{aufgabeBox}
Finde den korrekten Gerätepfad für dein USB-Gerät und
mache ihn stabil -- unabhängig von der Einsteckreihenfolge.
\end{aufgabeBox}

\begin{problemBox}
Linux vergibt Gerätepfade dynamisch beim Einstecken.
\texttt{ttyUSB0} kann morgen \texttt{ttyUSB1} sein -- je nachdem,
was zuerst eingesteckt wird.
\end{problemBox}

\begin{wissenBox}
Wir brauchen: \texttt{lsusb}, \texttt{dmesg}, das \texttt{/dev/}-Verzeichnis
und udev-Regeln für stabile Gerätezuordnung.
\end{wissenBox}

\section{Gerät identifizieren}

\begin{lstlisting}[style=bash]
# Alle USB-Geraete anzeigen
lsusb
# Ausgabe z.B.:
# Bus 001 Device 003: ID 2341:0043 Arduino SA Uno R3

# Geraet einstecken und Kernelmeldungen lesen
dmesg | tail -20
# Ausgabe z.B.:
# [12345.678] usb 1-1.4: new full-speed USB device
# [12345.890] cdc_acm 1-1.4:1.0: ttyACM0: USB ACM device

# Serielle Ports anzeigen
ls /dev/ttyUSB* /dev/ttyACM* 2>/dev/null

# Details zu einem Geraet
udevadm info /dev/ttyACM0 | grep -E "ID_SERIAL|ID_VENDOR|ID_MODEL"
\end{lstlisting}

\section{Stabile Gerätezuordnung mit udev}

\begin{lstlisting}[style=text]
# /etc/udev/rules.d/99-arduino.rules
# Arduino immer als /dev/arduino erreichbar machen

SUBSYSTEM=="tty", \
ATTRS{idVendor}=="2341", \
ATTRS{idProduct}=="0043", \
SYMLINK+="arduino", \
MODE="0666"
\end{lstlisting}

\begin{lstlisting}[style=bash]
# Regel aktivieren
sudo udevadm control --reload-rules
sudo udevadm trigger

# Pruefen
ls -la /dev/arduino
# -> /dev/arduino -> ttyACM0
\end{lstlisting}

\begin{loesungBox}
Nach der udev-Regel:
\begin{lstlisting}[style=python]
import serial
# Immer dieser Pfad -- egal ob ttyACM0 oder ttyACM1
arduino = serial.Serial('/dev/arduino', 9600, timeout=1)
\end{lstlisting}
Der Name \texttt{/dev/arduino} bleibt stabil.
\end{loesungBox}

\begin{merksatzBox}
udev-Regeln identifizieren Geräte über ihre USB-IDs (Vendor + Product),
nicht über den zufälligen Gerätepfad. So bekommt dasselbe physische
Gerät immer denselben Namen -- unverzichtbar für robuste Systeme.
\end{merksatzBox}

\begin{fehlerBox}
\begin{itemize}
  \item \textbf{Permission denied auf /dev/ttyUSB0:}\\
        \texttt{sudo usermod -aG dialout \$USER} und neu einloggen.
  \item \textbf{Falscher Pfad:} Immer mit \texttt{dmesg | tail} prüfen,
        welcher Pfad beim Einstecken vergeben wurde.
\end{itemize}
\end{fehlerBox}

\begin{uebungBox}
Stecke einen Arduino oder einen USB-Serial-Adapter an.
Finde mit \texttt{dmesg} und \texttt{lsusb} Vendor-ID und Product-ID.
Schreibe eine udev-Regel und teste sie.
\end{uebungBox}

\begin{quizBox}
\begin{enumerate}
  \item Warum ist \texttt{/dev/ttyUSB0} kein stabiler Pfad?
  \item Was speichern udev-Regeln zur Geräteidentifikation?
  \item Welcher Befehl zeigt Kernel-Meldungen beim Einstecken?
\end{enumerate}
\textbf{Antworten:} 1. Der Pfad wird dynamisch vergeben und ändert sich.
2. USB Vendor-ID und Product-ID (nicht den Pfad).
3. \texttt{dmesg | tail}
\end{quizBox}

\begin{haubeBox}
\textbf{udev} ist das Linux-Subsystem für Geräteverwaltung. Es läuft
als Daemon und reagiert auf Kernel-Events beim Einstecken/Entfernen.
Regeln können nicht nur Symlinks erzeugen, sondern auch Skripte
ausführen -- z.\,B. automatisch einen Treiber laden oder eine
Anwendung starten, wenn ein Gerät eingesteckt wird.
\end{haubeBox}

\begin{robotikBox}
ROS2-Treiber für Lidar, Kameras und IMUs nutzen udev-Regeln
standardmäßig. Das \texttt{ros-jazzy-lidar-driver}-Paket installiert
automatisch eine udev-Regel -- deshalb funktioniert der Lidar
sofort nach dem Einstecken, ohne manuelle Pfadkonfiguration.
\end{robotikBox}

\chapter{Wir wollen verstehen, wie ein Computer später einen Motor steuert}

\kapitelziel{
Du verstehst das Prinzip der PWM-Motorsteuerung und kannst
mit Python über Serial-Kommunikation einen Arduino ansprechen.
}
\kapitelstruktur

\section{Situation}
Du willst den linken Motor des Roboters mit 50\% Geschwindigkeit
vorwärts fahren lassen. Wie wird dieser Wunsch zur Spannung am Motor?

\begin{aufgabeBox}
Verstehe die Kette vom Python-Befehl über Serial und Arduino
bis zum Motorstrom.
\end{aufgabeBox}

\begin{wissenBox}
Wir brauchen: PWM-Grundlagen, die Serial-Kommunikation
zwischen Linux und Arduino und das L298N-Motortreiber-Prinzip.
\end{wissenBox}

\section{PWM -- Pulsweite steuert Leistung}

\begin{lstlisting}[style=text]
PWM (Pulse Width Modulation):
   _____     _____     _____
  |     |   |     |   |     |     50% Duty Cycle -> halbe Leistung
  |     |___|     |___|     |___

   ________  ________  ________
  |        ||        ||        |  80% Duty Cycle -> 80% Leistung
  |        ||        ||        |__
\end{lstlisting}

\section{Kommunikation: Linux → Arduino → Motor}

\begin{lstlisting}[style=python]
# motor_control.py -- Python auf Linux-Seite
import serial, time

class MotorController:
    """Sendet Fahrbefehle an Arduino via Serial."""
    
    def __init__(self, port='/dev/arduino', baudrate=9600):
        self.ser = serial.Serial(port, baudrate, timeout=1)
        time.sleep(2)  # Arduino reset-Zeit abwarten
    
    def fahre(self, links_pwm: int, rechts_pwm: int):
        """PWM-Werte 0-255, negativ = rueckwaerts."""
        befehl = f"M {links_pwm} {rechts_pwm}\n"
        self.ser.write(befehl.encode())
        antwort = self.ser.readline().decode().strip()
        return antwort
    
    def stopp(self):
        return self.fahre(0, 0)
    
    def __del__(self):
        if self.ser.is_open:
            self.stopp()
            self.ser.close()

# Verwendung
mc = MotorController('/dev/arduino')

print("Vorwaerts...")
mc.fahre(150, 150)
time.sleep(2)

print("Kurve links...")
mc.fahre(50, 150)
time.sleep(1)

print("Stopp.")
mc.stopp()
\end{lstlisting}

\begin{lstlisting}[style=text]
// Arduino-Seite (vereinfacht)
// Empfängt "M links rechts\n" und steuert L298N

void loop() {
  if (Serial.available()) {
    String cmd = Serial.readStringUntil('\n');
    if (cmd.startsWith("M ")) {
      int l = cmd.substring(2, cmd.indexOf(' ', 2)).toInt();
      int r = cmd.substring(cmd.lastIndexOf(' ')+1).toInt();
      setMotors(l, r);
      Serial.println("OK");
    }
  }
}
\end{lstlisting}

\begin{loesungBox}
Die vollständige Kette:
\begin{enumerate}
  \item Python: \texttt{mc.fahre(150, 150)} → String \texttt{"M 150 150\textbackslash n"}
  \item Serial: Bytes über USB-Kabel zum Arduino
  \item Arduino: parst Befehl, setzt PWM-Signal
  \item L298N: verstärkt PWM zu 12V-Motorstrom
  \item DC-Motor: dreht mit proportionaler Geschwindigkeit
\end{enumerate}
\end{loesungBox}

\begin{merksatzBox}
Der Raspberry Pi (oder jeder Linux-Computer) übernimmt die
\textit{Intelligenz}: Planung, Navigation, Entscheidungen.
Der Arduino übernimmt das \textit{Echtzeit-Timing}: PWM-Generierung,
Encoder-Auswertung, Notabschaltung. Diese Trennung ist bewusst
und professionell.
\end{merksatzBox}

\begin{robotikBox}
In Band 2 ersetzen wir den direkten Serial-Kanal durch einen
ROS2-Node. Der Motor-Controller-Node subscribt auf
\texttt{/cmd\_vel} (geometry\_msgs/Twist) und konvertiert
Geschwindigkeitsbefehle in PWM-Werte -- dasselbe Prinzip,
aber eingebettet in die ROS2-Infrastruktur.
\end{robotikBox}
